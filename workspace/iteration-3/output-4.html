Why We Migrated From REST to GraphQL

Author
Priya Sharma
Published
November 12, 2025

After four years of building our platform on a REST API, we made the decision to migrate to GraphQL. It wasn't an overnight choice — it came after months of growing pains, developer frustration, and a mounting list of performance complaints from our mobile clients. Here's an honest account of what drove us to make the switch, how we approached the migration, and what the numbers looked like on the other side.

The Problem

Our REST API had grown organically over the years. What started as a clean, well-documented set of endpoints gradually became a sprawling collection of over 200 routes, many of them overlapping in purpose or returning more data than any single client actually needed.

The most painful symptom was over-fetching. Our mobile app's dashboard screen required data from six separate endpoints — user profile, recent activity, notifications, team members, project summaries, and billing status. Six round trips on a 4G connection meant users waited up to three seconds before they saw anything meaningful. We tried batching calls on the server, but it felt like papering over a deeper structural issue.

Under-fetching was the other side of the coin. For our web client, displaying a project card meant hitting the projects endpoint, then making follow-up requests for owner details and the latest activity because the original response didn't include them. Our frontend engineers had grown accustomed to writing request waterfalls as a matter of course.

Version management made things worse. We were simultaneously maintaining /v1, /v2, and a private /v3-beta that only two internal teams were using. Every schema change risked breaking one of the three versions, and deprecating old endpoints was a months-long negotiation.

"Six round trips on a 4G connection meant users waited up to three seconds before they saw anything meaningful."

Our Approach

We didn't do a big-bang rewrite. Instead, we adopted a strangler fig strategy — introducing GraphQL alongside our existing REST API and gradually moving traffic over as each domain area was ready.

We chose Apollo Server running on Node.js, primarily because our backend team already had strong TypeScript experience and the ecosystem fit well. We started by defining the schema for the user and project domains, which together accounted for roughly 60% of our API traffic.

One early decision that paid off was enabling Apollo's response caching from the start. By annotating fields with cache hints in the schema, we got significant cache hit rates without any client-side changes: 

type Project @cacheControl(maxAge: 60) {
  id: ID!
  name: String!
  owner: User @cacheControl(maxAge: 300)
  createdAt: String!
  updatedAt: String!
  members: [User!]!
  recentActivity: [ActivityEvent!]! @cacheControl(maxAge: 10)
}

We ran both APIs in parallel for eight months. During that time, any team shipping a new frontend feature was expected to use the GraphQL endpoint. Existing REST consumers were not forced to migrate immediately, but new work had to go through GraphQL. This gave us a steady, low-pressure migration path rather than a hard cutover.

Results

Twelve months after beginning the migration and six months after shutting down the last REST endpoint, the results have been meaningful and measurable.

Dashboard load time on mobile dropped from an average of 2.8 seconds to 0.9 seconds — a 68% reduction. This came almost entirely from collapsing six network requests into one and eliminating the over-fetched fields that were slowing serialisation. Our 95th percentile time-to-interactive improved by a similar margin.

Developer satisfaction also shifted noticeably. In our quarterly survey, frontend engineers rated the API experience 4.3 out of 5, up from 2.7 before the migration. The most cited reasons were "I only get the data I ask for" and "the schema tells me everything I need to know — I don't need to read the docs to discover fields."

We did hit some rough patches. N+1 query problems were initially worse than with REST — our naive resolver implementation caused a flood of database queries that we hadn't anticipated. We solved this with DataLoader, but it took two sprints longer than we'd planned. If we were starting again, we'd introduce DataLoader from day one rather than retrofitting it after performance issues surfaced in production.

On balance, the migration was worth the effort. The codebase is more maintainable, our mobile clients are faster, and onboarding new frontend engineers takes noticeably less time because the schema serves as living, queryable documentation.
