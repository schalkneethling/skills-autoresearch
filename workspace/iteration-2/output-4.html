Why We Migrated From REST to GraphQL

Author
Priya Sharma
Published
November 12, 2025
Topics
GraphQL
API Design
Backend

After three years of building on a REST API that served us well in the early days, our engineering team made the decision to migrate our primary data layer to GraphQL. It wasn't a decision we made lightly — we had hundreds of client integrations, a well-understood codebase, and engineers who knew REST inside out. This is the story of why we made that call, how we approached it, and what we learned.

The Problem

Our REST API had grown organically over three years, and the cracks were starting to show. Mobile clients were particularly affected: a single screen in our iOS app required up to seven sequential API requests, each one dependent on data from the last. Network latency on slower connections turned routine interactions into frustrating waits.

Over-fetching was rampant. Our /users/{id} endpoint returned 47 fields — but the average client used fewer than ten. We were serialising, transmitting, and deserialising data that would never be read. On high-traffic endpoints this added up to measurable infrastructure cost and battery drain on mobile devices.

The other side of the coin was under-fetching: our product teams frequently needed data relationships that required multiple round trips. Every new feature triggered a conversation about whether to add a new endpoint, extend an existing one, or tell the client team to make two requests. None of those options felt right.

"Every new feature triggered a conversation about whether to add a new endpoint, extend an existing one, or tell the client to make two requests. None of those options felt right."

Our Approach

We didn't rewrite everything at once. Instead, we ran GraphQL alongside our existing REST API using a strangler fig pattern. New product features were built exclusively against the GraphQL layer, while legacy REST endpoints remained available to existing integrations. This let us validate our schema design with real product work before committing to a full migration.

We used Apollo Server and built our schema collaboratively with frontend and mobile engineers — the people who would actually be writing queries. This shifted the design conversation from "what should this endpoint return?" to "what does the client actually need?" The difference in framing produced better outcomes.

Our resolver architecture used DataLoader for batching and caching, which solved the N+1 query problem we'd been worried about. A simplified example of how we structured a user resolver: 

// userResolver.js
const { createBatchLoader } = require('./loaders');

const userLoader = createBatchLoader(async (ids) => {
  const users = await db.users.findMany({
    where: { id: { in: ids } },
  });
  // Return in the same order as requested IDs
  return ids.map((id) => users.find((u) => u.id === id) ?? null);
});

const resolvers = {
  Query: {
    user: (_, { id }) => userLoader.load(id),
  },
  User: {
    posts: (user) => postLoader.load(user.id),
    followers: (user) => followerLoader.load(user.id),
  },
};

We also invested heavily in schema governance from day one. A custom linting ruleset enforced naming conventions, required descriptions on all types and fields, and prevented breaking changes from landing on the main branch without a deprecation notice.

Results

Six months after completing the migration, the numbers told a clear story. The average number of network requests per screen on iOS dropped from 4.3 to 1.1. Response payload sizes fell by an average of 62% — not because we were compressing more aggressively, but because clients were simply requesting less.

Developer velocity improved in ways that are harder to quantify but easier to feel. Our frontend engineers stopped waiting on backend changes to build new views. The schema became a shared contract that both sides trusted, and the number of cross-team "clarification" meetings dropped noticeably.

We also saw infrastructure savings: lower payload sizes and reduced request counts translated directly to lower egress costs and a reduction in average server response time. GraphQL wasn't a silver bullet — our schema governance overhead was real, and onboarding new engineers took longer initially — but for our use case, the tradeoffs were firmly in our favour.
