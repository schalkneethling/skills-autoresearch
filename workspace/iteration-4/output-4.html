Why We Migrated From REST to GraphQL

Author
Priya Sharma
Published
November 12, 2025

After three years of shipping a REST API that served our web app, mobile clients, and a growing list of third-party integrations, we hit a wall. Response payloads were bloated, client teams were filing weekly tickets asking for new endpoints, and our backend engineers were spending more time writing boilerplate than solving real problems. In late 2024 we decided to migrate to GraphQL — and six months on, we haven't looked back.

The Problem

Our REST API had grown organically over three years into a maze of over 200 endpoints. The web dashboard needed a summary view that pulled data from nine separate endpoints; the iOS app needed a slimmer version of the same data but without the nested relationships. Every new screen request from product turned into a two-week back-and-forth negotiation between frontend and backend teams about what shape the response should take.

Over-fetching was rampant. A call to /api/v2/projects returned full project objects — including deeply nested member lists and activity feeds — even when a component only needed the project name and last-modified date. On mobile, this translated directly into wasted bandwidth and slower perceived performance in markets where connectivity is unreliable.

Under-fetching was equally painful. The project detail page required four round trips before it could render meaningful content: one for the project, one for its members, one for the open issue count, and one for the latest activity. Waterfall loading was the norm, not the exception.

"Every new screen request turned into a two-week negotiation about what shape the response should take."

Our Approach

Rather than a big-bang cutover, we ran GraphQL alongside our existing REST API for four months. We used Apollo Server on top of our existing Node.js services, resolving fields against the same data layer the REST handlers already called. This meant no database schema changes and no service rewrites — just a new translation layer.

Our first schema focused on the three highest-traffic resource types: projects, users, and issues. We defined the types collaboratively — frontend leads wrote the queries they wished they could make, and backend engineers reviewed feasibility and flagged N+1 risks early. DataLoader batching was added before we shipped to production, not as an afterthought.

Here is the simplified type definition that replaced our three most-requested REST endpoints: 

type Project {
  id: ID!
  name: String!
  description: String
  members(limit: Int = 10): [User!]!
  openIssueCount: Int!
  lastActivityAt: DateTime!
}

type User {
  id: ID!
  displayName: String!
  avatarUrl: String
  role: ProjectRole!
}

type Query {
  project(id: ID!): Project
  projects(ownerId: ID, archived: Boolean): [Project!]!
}

With this schema, the dashboard summary view went from nine REST calls to a single GraphQL query that fetched exactly the fields it needed. The iOS app sent the same query but requested fewer fields — no new endpoint, no new ticket, no waiting.

Results

Six months after migrating our first client to GraphQL, the numbers were clear. Average payload size dropped by 62% for the dashboard and 78% for the mobile app. The time-to-first-meaningful-paint on our project detail page fell from 1.8 seconds to 640 milliseconds — a reduction of over 60% — because we eliminated the waterfall of sequential REST requests.

Backend engineer time spent on new-endpoint requests dropped from roughly 30% of sprint capacity to under 5%. Frontend teams now add new views without waiting on a backend release; they query for the shape they need and the resolver handles it.

There were real costs too. Schema design requires upfront discipline. Our first attempt at a mutation API was too coarse-grained and had to be partially redesigned after three months. Caching is also genuinely harder — we moved from simple HTTP caching to a combination of Apollo Client's normalized cache and server-side persisted queries. Neither was free to implement. But for a product with a broad and evolving API surface, the trade-offs have been worth it.
