Why We Migrated From REST to GraphQL

Author
Priya Sharma
Published
12 November 2025
Topic
API Design

Last year our mobile app made an average of 14 API calls per screen render. Users on slow connections saw spinners where there should have been content, and our backend team spent more time writing one-off endpoints than solving real problems. After six months of incremental pain, we made the decision to migrate our primary data layer from REST to GraphQL — and it changed how our entire engineering organisation thinks about data fetching.

"After six months of incremental pain, we made the decision to migrate our primary data layer from REST to GraphQL — and it changed how our entire engineering organisation thinks about data fetching."

The Problem

Our REST API had grown organically over four years. What started as a clean, resource-oriented design had accumulated dozens of bespoke endpoints built to satisfy specific screen requirements. The /dashboard/summary endpoint returned 47 fields; most clients used fewer than ten of them. Meanwhile, the mobile team regularly filed tickets asking for new endpoints because combining existing ones required either multiple round trips or a backend change that took two weeks to ship.

Over-fetching wasted bandwidth. Under-fetching forced waterfalls. The feedback loop between frontend need and backend delivery was too slow. We weren't moving as fast as we needed to, and our API surface was becoming impossible to document consistently.

Our Approach

We decided against a big-bang rewrite. Instead, we introduced a GraphQL layer that sat in front of our existing REST services, translating queries into REST calls while we migrated underlying resolvers to hit the database directly over time. This gave us a safe incremental path without disrupting clients mid-flight.

We used Apollo Server with a schema-first approach. The team spent two weeks writing the schema before a single resolver was implemented — this turned out to be one of the most valuable investments. It forced cross-team alignment on naming, nullability, and what data relationships actually looked like.

Our schema entry point for user profile data looked like this: 

type Query {
  userProfile(id: ID!): UserProfile
}

type UserProfile {
  id: ID!
  displayName: String!
  email: String!
  avatarUrl: String
  createdAt: String!
  membershipTier: MembershipTier!
  recentActivity(limit: Int = 5): [ActivityItem!]!
}

enum MembershipTier {
  FREE
  PRO
  ENTERPRISE
}

type ActivityItem {
  id: ID!
  type: String!
  description: String!
  occurredAt: String!
}

Defining recentActivity as a nested type with its own arguments meant that a mobile client showing a slim profile card could query just displayName, avatarUrl, and membershipTier — no wasted bytes.

Results

Three months after the full rollout, the numbers were unambiguous. The average number of network requests per screen dropped from 14 to 3. Median page load time on our mobile web app fell by 38%. The backend team closed 23 open tickets for bespoke endpoint requests — they were no longer needed.

Perhaps more importantly, the feedback loop between product and engineering shortened dramatically. Frontend developers could prototype new screens using existing schema types without waiting for backend work. When a new field was genuinely needed, the schema review process took a day rather than a sprint.

There are trade-offs. GraphQL queries are harder to cache at the network level than REST responses, and we had to invest in query complexity analysis to protect against expensive client-constructed queries. Tooling overhead — schema stitching, code generation, persisted queries — is real. For teams with simple data requirements and stable clients, REST remains perfectly adequate. But for us, the migration was the right call.
